# KoalaFlyff — AI Coding Context & Guidelines

Welcome, AI Coding Agent! This file outlines the architectural constraints, critical rules, and state paradigms of the **KoalaFlyff** extension. Read this document thoroughly before proposing or implementing any changes to the codebase.

---

## 🏛️ System Architecture Invariants

KoalaFlyff is a Manifest V3 browser extension. It consists of:
1. **`popup/` (UI):** High-density panel layout (Settings, History, Logs). Runs in a standard UI window context.
2. **`background.js` (Mediation):** Manifest V3 Background Service Worker.
3. **`content/content.js` (Injection):** Intercepts inputs on Flyff Universe tabs and programmatically simulates matching keyboard events on the target game canvas.

---

## ⚠️ CRITICAL RULES (Never Violate)

### 1. The Ephemeral Service Worker Lifecycle
- **Constraint:** In Manifest V3, the background service worker (`background.js`) is suspended by Chrome after ~30 seconds of inactivity. It is woken up on-demand when a new event (message, tab removal, etc.) is triggered.
- **Rule:** Global memory variables (like `config`, `sessionStartTime`, `systemLogs`) are completely lost when suspended. While we restore them asynchronously from storage, **all event listeners MUST await the `initPromise` before executing any logic.**
- **Implementation Pattern:**
  ```javascript
  // Always register the initPromise globally
  let initPromise = initialize();

  chrome.tabs.onRemoved.addListener(async (tabId) => {
    await initPromise; // Crucial!
    // ... logic ...
  });

  chrome.runtime.onMessage.addListener((message, sender, sendResponse) => {
    if (message.type === 'SOME_MESSAGE') {
      initPromise.then(() => {
        // ... logic ...
        sendResponse(result);
      });
      return true; // CRITICAL: keeps channel open for async sendResponse
    }
    return false;
  });
  ```

### 2. Recursive Keypress Loop Prevention
- **Constraint:** We capture keydowns in the content script and dispatch simulated keydowns back to Flyff tabs.
- **Rule:** If we capture simulated events, the extension will enter an infinite loop of routing its own synthetic inputs, causing browser freeze/crash.
- **Rule:** **You MUST NEVER remove, bypass, or weaken the `!e.isTrusted` check in `content/content.js`.** Only physical keystrokes generated by a human operator may be captured and routed.
- **Implementation Pattern:**
  ```javascript
  window.addEventListener('keydown', (e) => {
    if (!e.isTrusted) return; // Prevent infinite simulation feedback loops
    // ...
  }, true);
  ```

### 3. Key Simulation Guard
- **Constraint:** Key mapping rows in the popup settings can be added with empty target fields (`keyB = ""`).
- **Rule:** Never attempt to simulate an empty string. Doing so causes `NaN` assignments to event `keyCode` and `which` properties, triggering errors.
- **Implementation Pattern:**
  ```javascript
  function simulateKeyPress(keyChar) {
    if (!keyChar) return; // Guard clause
    // ... event simulation ...
  }
  ```

### 4. Storage Separation
- **`chrome.storage.local`:** Holds durable, cross-session configurations (`multiBoxConfig`) and playtime records (`sessionHistory`).
- **`chrome.storage.session`:** Holds ephemeral, in-memory session information (`sessionStartTime`, `systemLogs`) that should be wiped clean when the browser closes.

## 🛠️ API & Message Passing Specification

All communications are mediated through Chrome runtime messages. Maintain this strict protocol:

| Message Type | Sender | Receiver | Payload | Response | Description |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `REQUEST_CONFIG` | Content / Popup | Background | *None* | `{ ...config, sessionStartTime }` | Fetches active worker config. |
| `UPDATE_CONFIG` | Popup | Background | `{ payload: config }` | `{ status: "success" }` | Updates and broadcasts configuration. |
| `CONFIG_CHANGED` | Background | Content | `{ config }` | *None* | Broadcasts new state to active tabs. |
| `KEY_PRESSED` | Content | Background | `{ key: String }` | *None* | Reports mapped keypress from listener. |
| `SIMULATE_KEY` | Background | Content | `{ key: String }` | *None* | Requests key simulation on target tab. |
| `ADD_LOG` | Content / Popup | Background | `{ log: String, timestamp: Number }` | *None* | Appends a diagnostic entry. |
| `REQUEST_LOGS` | Popup | Background | *None* | `Array<LogEntry>` | Retrieves active worker logs. |
| `TOGGLE_SERVICE_REQUEST` | Content | Background | *None* | *None* | Instantly toggles the worker state (hotkey). |
